🎯 Introducción a k6: arquitectura interna

k6 es una herramienta de pruebas de carga de código abierto desarrollada por Grafana Labs, escrita en Go y con una API en JavaScript ES6 (con soporte nativo de TypeScript desde k6 1.0). A diferencia de JMeter o LoadRunner, k6 fue diseñado para ser eficiente, moderno y fácil de integrar en pipelines CI/CD. Cada usuario virtual (VU) corre en una goroutine, consumiendo muy pocos recursos — puedes simular miles de VUs en una sola máquina.

¿Por qué k6 y no otro?

🔧 Componentes clave de un script k6:
  • Inicialización (init): se ejecuta una sola vez por VU al arrancar (imports, open(), SharedArray).
  • Setup: función opcional ejecutada una vez antes del test (ideal para login, semilla de datos).
  • Función default (VU code): ejecutada por cada VU repetidamente durante el test.
  • Teardown: función opcional ejecutada una vez al finalizar (limpieza).
  • Opciones: stages, thresholds, escenarios, tags globales — declarativas.
💡 k6 no es un navegador: las peticiones HTTP son a nivel de protocolo, lo que lo hace extremadamente rápido para APIs. Para pruebas de front-end real con renderizado se usa el módulo k6/browser (basado en Chromium DevTools Protocol).

📝 Primer script completo con opciones y fases

Anatomía de un script k6 profesional: imports, métricas personalizadas, opciones, setup, default y teardown.

import http from 'k6/http';
import { check, sleep } from 'k6';
import { Rate } from 'k6/metrics';

// ──────────────────────────────────────────────────────────
// 1. Inicialización (se ejecuta una vez por VU al cargar)
// ──────────────────────────────────────────────────────────
const errores = new Rate('errores_total');

// ──────────────────────────────────────────────────────────
// 2. Opciones del test (declarativas)
// ──────────────────────────────────────────────────────────
export const options = {
    stages: [
        { duration: '1m', target: 50 },   // rampa hasta 50 VUs
        { duration: '2m', target: 50 },   // mantener 50 VUs
        { duration: '30s', target: 0 },   // degradado
    ],
    thresholds: {
        http_req_duration: ['p(95) < 500', 'p(99) < 800'],
        http_req_failed:   ['rate < 0.01'],
        errores_total:     ['rate < 0.05'],
    },
    tags: { environment: 'staging', team: 'backend' },
};

// ──────────────────────────────────────────────────────────
// 3. Setup: se ejecuta UNA VEZ antes de las iteraciones
// ──────────────────────────────────────────────────────────
export function setup() {
    const res = http.post('https://reqres.in/api/login', JSON.stringify({
        email: 'eve.holt@reqres.in',
        password: 'cityslicka',
    }), { headers: { 'Content-Type': 'application/json' } });

    return { authToken: res.json('token') };
}

// ──────────────────────────────────────────────────────────
// 4. Función default: la ejecuta cada VU en cada iteración
// ──────────────────────────────────────────────────────────
export default function (data) {
    const headers = { 'Authorization': `Bearer ${data.authToken}` };
    const res = http.get('https://reqres.in/api/users/2', { headers });

    const success = check(res, {
        'status es 200':   (r) => r.status === 200,
        'body tiene data': (r) => r.json('data') !== undefined,
    });
    errores.add(!success);

    sleep(1);  // think time
}

// ──────────────────────────────────────────────────────────
// 5. Teardown: limpieza final
// ──────────────────────────────────────────────────────────
export function teardown(data) {
    console.log(`Test finalizado. Token usado: ${data.authToken}`);
}
📌 setup/teardown permiten autenticación, creación de datos de prueba y limpieza posterior — son fundamentales para entornos reales donde necesitas dejar el sistema limpio.

🔄 Ciclo de vida de un test k6

Entender el orden de ejecución es esencial para evitar errores como llamar open() dentro de default o crear métricas en mal sitio.

┌─────────────────────────────────────────────────┐
│  1. INIT (una vez por VU, al arrancar)          │
│     - imports                                   │
│     - declarar métricas custom                  │
│     - open() de archivos                        │
│     - SharedArray                               │
│     - export const options = { ... }            │
└─────────────────────────────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────┐
│  2. SETUP (una vez, en VU especial)             │
│     - autenticación maestra                     │
│     - seed de datos                             │
│     - return data → pasa a default y teardown   │
└─────────────────────────────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────┐
│  3. DEFAULT (cada VU, en bucle, durante test)   │
│     - http requests                             │
│     - checks, sleep                             │
│     - métricas dinámicas                        │
└─────────────────────────────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────┐
│  4. TEARDOWN (una vez, al finalizar)            │
│     - cleanup de datos creados                  │
│     - notificaciones                            │
└─────────────────────────────────────────────────┘
⚠️ Errores típicos por ignorar el ciclo:
  • Llamar open('./data.csv') dentro de default() → ¡no funciona! Solo en INIT.
  • Hacer requests pesados en setup sin medir el coste — su duración cuenta para el tiempo total del test.
  • Crear métricas (new Rate(...)) dentro de default → error.
  • Compartir estado mutable entre VUs vía variables globales → cada VU tiene su propia VM JS, no se ven entre sí.

Variables especiales disponibles en runtime

VariableSignificado
__VUID del usuario virtual actual (1-N). Es 0 durante init.
__ITERNúmero de iteración del VU actual (empieza en 0).
__ENVObjeto con todas las variables de entorno disponibles.

exec.test.optionsAcceso runtime a las opciones configuradas.

✅ Checks y Thresholds: validaciones robustas

La distinción entre checks y thresholds es uno de los conceptos más importantes en k6:

ChecksThresholds
¿Detienen el test?No (informativos)Sí (pueden abortarlo con abortOnFail)
¿Afectan exit code?No por defectoSí — exit 99 si fallan
GranularidadPor iteraciónAgregado global de métricas
Mejor paraValidar respuestas individualesDefinir SLOs del test

Checks básicos

import http from 'k6/http';
import { check } from 'k6';

export default function () {
    const res = http.get('https://test-api.k6.io/public/crocodiles/1/');

    check(res, {
        'status 200':       (r) => r.status === 200,
        'tiempo < 500ms':   (r) => r.timings.duration < 500,
        'tiene id':         (r) => r.json('id') !== undefined,
        'es JSON válido':   (r) => r.headers['Content-Type'].includes('application/json'),
        'body no vacío':    (r) => r.body.length > 0,
    });
}

Thresholds avanzados con filtros por tag

export const options = {
    thresholds: {
        // Globales
        http_req_failed:   ['rate < 0.01'],
        http_reqs:         ['count > 1000'],
        checks:            ['rate > 0.95'],

        // Percentiles + media
        http_req_duration: ['p(95) < 400', 'p(99) < 800', 'avg < 200'],

        // Filtros por tag (sub-métricas)
        'http_req_duration{type:critical}': ['p(90) < 300'],
        'http_req_duration{endpoint:login}': ['p(95) < 500'],
        'checks{type:critical}': ['rate > 0.99'],

        // Por escenario
        'http_req_duration{scenario:spike}': ['p(95) < 2000'],

        // Con abortOnFail (corta el test si se viola)
        http_req_duration: [
            { threshold: 'p(95) < 500', abortOnFail: true, delayAbortEval: '30s' },
        ],
    },
};

export default function () {
    http.get('https://api.example.com/users', {
        tags: { type: 'critical', endpoint: 'users' },
    });
}

Checks que afectan el exit code

Por defecto un check fallido no hace fallar la build de CI. Para que sí lo haga, conviértelo en threshold:

export const options = {
    thresholds: {
        // Falla la build si menos del 99% de checks pasan
        checks: [{ threshold: 'rate >= 0.99', abortOnFail: false }],
    },
};
💡 Truco profesional: agrupa checks por dominio funcional con group() y aplica thresholds por grupo: 
'group_duration{group:::Checkout}': ['p(95) < 800'],
'checks{group:::Login}': ['rate > 0.99'],

🎯 Assertions API k6 2.0

k6 2.0 introduce una Assertions API más expresiva, inspirada en Jest/Playwright. Se diferencia de check porque las aserciones pueden detener la iteración (estilo testing tradicional) y producen mensajes de error legibles.

import { expect } from 'k6/expect';
import http from 'k6/http';

export default function () {
    const res = http.get('https://api.example.com/products/42');

    // Estilo fluent
    expect(res.status).toBe(200);
    expect(res.timings.duration).toBeLessThan(500);
    expect(res.json('name')).toBe('Producto X');
    expect(res.json('tags')).toContain('promo');
    expect(res.json('price')).toBeGreaterThan(0);

    // Negaciones
    expect(res.json('deleted')).not.toBe(true);

    // Soft assertions: no detienen pero registran el fallo
    expect.soft(res.headers['Content-Type']).toContain('json');
}
ℹ️ La Assertions API convive con check(). Usa checks cuando quieres seguir iterando aunque falle, y expect cuando un fallo debe abortar la iteración (test funcional). En la consola, los fallos de expect incluyen el valor recibido y el esperado — mucho más útil para debugging.

📊 Métricas personalizadas y etiquetas dinámicas

k6 ofrece cuatro tipos de métricas custom:

import http from 'k6/http';
import { Counter, Gauge, Rate, Trend } from 'k6/metrics';

// Declarar las métricas (siempre en init, no dentro de default)
const pedidosCreados   = new Counter('pedidos_creados');
const tamanoUltimo     = new Gauge('tamano_respuesta_kb');
const tasaCheckout     = new Rate('checkout_exitoso');
const duracionDB       = new Trend('db_query_time', true);  // true → como duración

export default function () {
    const inicio = Date.now();
    const res = http.post('https://api.tienda.com/checkout', JSON.stringify({ items: 3 }));
    const ms = Date.now() - inicio;

    // Counter (sólo suma)
    if (res.status === 201) pedidosCreados.add(1);

    // Gauge (último valor)
    tamanoUltimo.add(res.body.length / 1024);

    // Rate (true/false)
    tasaCheckout.add(res.status === 201);

    // Trend (estadísticas)
    duracionDB.add(ms, { endpoint: 'checkout' });   // con tag
}

📂 Carga masiva de datos desde CSV y JSON

Para simular múltiples usuarios únicos sin colisiones, usa SharedArray — carga el dataset una sola vez en memoria compartida entre todos los VUs.

CSV simple (split manual)

import { SharedArray } from 'k6/data';
import http from 'k6/http';

const usuarios = new SharedArray('credenciales', function () {
    const data = open('./usuarios.csv');
    return data.split('\n')
        .filter(line => line.trim().length > 0)
        .map(line => {
            const [email, pass] = line.split(',');
            return { email, pass };
        });
});

export default function () {
    const user = usuarios[(__VU + __ITER) % usuarios.length];  // determinista
    http.post('https://api.example.com/login', JSON.stringify({
        email: user.email,
        password: user.pass,
    }));
}

CSV con headers (PapaParse)

import { SharedArray } from 'k6/data';
import papaparse from 'https://jslib.k6.io/papaparse/5.1.0/index.js';

const usuarios = new SharedArray('users', function () {
    const csv = open('./100k_usuarios.csv');
    return papaparse.parse(csv, { header: true, dynamicTyping: true }).data;
});

export default function () {
    // VU N usa siempre el usuario N (determinismo total)
    const user = usuarios[__VU % usuarios.length];
    // ...
}

JSON estructurado

const productos = new SharedArray('productos', () => {
    return JSON.parse(open('./productos.json'));
});
📌 NOTA: open() solo funciona en fase de inicialización (fuera de la función default). Para archivos grandes (>1GB) considera leer en fragmentos o usar SQL directamente con xk6-sql.
⚠️ No abuses de open() sin SharedArray: sin él, cada VU carga su propia copia del archivo en memoria. Con 500 VUs y un CSV de 50MB, son 25GB. SharedArray mantiene una sola copia.

🗃️ Data-driven avanzado: distribución sin colisiones

Cuando cada usuario debe ser único (sin re-login del mismo email), necesitas una estrategia de distribución determinista.

Estrategia 1: módulo VU+ITER (sin repeticiones inmediatas)

const usuarios = new SharedArray('users', loader);

export default function () {
    const idx = (__VU - 1) * exec.test.options.iterations + __ITER;
    const user = usuarios[idx % usuarios.length];
    // ...
}

Estrategia 2: pool global con índice atómico (xk6-counter)

// Necesita compilar con xk6-counter
import counter from 'k6/x/counter';
const usuarios = new SharedArray('users', loader);

export default function () {
    const idx = counter.up('users-idx') - 1;   // único globalmente
    const user = usuarios[idx % usuarios.length];

4. Refresh token automático (dentro del VU)

Cuando el token expira en mitad del test, hay que renovarlo. Estrategia: guardar en variable global por VU y refrescar al detectar 401:

let cachedToken = null;
let cachedExpiry = 0;

function getValidToken() {
    if (cachedToken && Date.now() < cachedExpiry - 30_000) return cachedToken;
    // refrescar
    const res = http.post('https://auth.example.com/refresh', JSON.stringify({
        refresh_token: __ENV.REFRESH_TOKEN,
    }), { headers: { 'Content-Type': 'application/json' }, tags: { name: 'refresh-token' } });
    cachedToken  = res.json('access_token');
    cachedExpiry = Date.now() + res.json('expires_in') * 1000;
    return cachedToken;
}

export default function () {
    let res = http.get('https://api.example.com/data', {
        headers: { Authorization: `Bearer ${getValidToken()}` },
    });
    if (res.status === 401) {
        cachedToken = null;
        res = http.get('https://api.example.com/data', {
            headers: { Authorization: `Bearer ${getValidToken()}` },
        });
    }
}

        await page.waitForTimeout(1000);

        // Métricas de performance
        const perf = await page.evaluate(() => JSON.stringify(performance.getEntriesByType('navigation')));
        console.log('Performance:', perf);
    } finally {
        await page.close();
    }
}
⚠️ El modo browser consume mucho más recursos que el modo HTTP puro. Úsalo con pocos VUs (5-20) y prefiérelo para flujos críticos de UX, no para carga masiva.
ℹ️ Compatibilidad con Playwright: k6 2.0 amplió la compatibilidad con la API de Playwright, lo que facilita migrar tests existentes. Más del 80% de los selectores y métodos comunes funcionan idénticamente.

import { browser } from 'k6/browser';

    scenarios: {

            rate: 500, timeUnit: '1s',
            duration: '5m',
            preAllocatedVUs: 100,
            maxVUs: 300,
            exec: 'apiLoad',
        },
        // 10%: navegadores reales validando UX
        frontend_ux: {
            executor: 'per-vu-iterations',
            vus: 5, iterations: 20,
            maxDuration: '5m',
            options: { browser: { type: 'chromium' } },
            exec: 'frontendCheck',
        },
    },
    thresholds: {
        // El frontend NO debería degradarse aunque el backend esté al límite
        'browser_web_vital_lcp{scenario:frontend_ux}': ['p(95) < 3000'],
        'http_req_duration{scenario:backend_load}':    ['p(95) < 1000'],
    },
};

export function apiLoad() {
    http.get(`${__ENV.BASE_URL}/api/products?page=${Math.floor(Math.random() * 50)}`);
}

export async function frontendCheck() {
    const page = await browser.newPage();
    try {
        await page.goto(`${__ENV.BASE_URL}/products`);
        await page.waitForSelector('.product-card');
        check(await page.locator('.product-card').count(), {
            'al menos 12 productos visibles': (n) => n >= 12,
        });
    } finally {
        await page.close();
    }
}
🎯 Por qué este patrón es valioso:
  • Los tests puramente API no detectan problemas de renderizado, lazy-load, hidratación SSR.
  • Los tests puramente browser no escalan para reproducir tráfico realista.
  • El híbrido valida la experiencia real bajo el peor escenario.

// helpers/auth.ts
import http from 'k6/http';

let cachedToken: string | null = null;
let cachedExpiry = 0;

export function getToken(): string {
    if (cachedToken && Date.now() < cachedExpiry - 30_000) return cachedToken;
    const res = http.post(`${__ENV.AUTH_URL}/login`, JSON.stringify({
        user: __ENV.USER, pass: __ENV.PASS,
    }), { headers: { 'Content-Type': 'application/json' } });
    cachedToken  = res.json('token') as string;
    cachedExpiry = Date.now() + (res.json('expires_in') as number) * 1000;
    return cachedToken;
}

// flows/checkout.ts
import http from 'k6/http';

import { getToken } from '../helpers/auth.ts';

export function checkout(productId: number): void {

        const res = http.post(`${__ENV.BASE_URL}/cart`,
            JSON.stringify({ productId }), { headers });
        check(res, { 'cart OK': (r) => r.status === 201 });
    });
}

🔗 Correlación y datos dinámicos

La correlación es extraer un valor de una respuesta para usarlo en la siguiente (p.ej. CSRF token, sessionId, IDs creados). Es uno de los pilares de cualquier test realista.

1. Extraer de un body JSON

const res = http.post(BASE + '/products', JSON.stringify({ name: 'X' }));
const productId = res.json('id');           // path simple
const tags      = res.json('tags');         // array completo
const nestedVal = res.json('meta.author.email');  // path con puntos

2. Extraer con jsonpath complejo (jslib)

import jsonpath from 'https://jslib.k6.io/jsonpath/1.0.2/index.js';

const all = jsonpath.value(res.json(), '$..items[?(@.active==true)].id');

3. Extraer con regex de un HTML (cuando no hay API)

const res = http.get('https://web.example.com/login');
const csrfMatch = res.body.match(/name="csrf_token" value="([^"]+)"/);
if (csrfMatch) {
    const csrf = csrfMatch[1];
    http.post('https://web.example.com/login', {
        username: 'admin', password: '123', csrf_token: csrf,
    });
}

4. Extraer con HTML parser (jslib/html-parser)

const doc = res.html();
const csrf = doc.find('input[name="csrf_token"]').first().attr('value');
const productIds = doc.find('.product-card').map((i, el) => el.attr('data-id')).toArray();

5. Generar datos dinámicos únicos

💣 Chaos engineering con xk6-disruptor

xk6-disruptor permite inyectar fallas en servicios Kubernetes desde tus scripts k6: añadir latencia, devolver errores HTTP, matar pods. Combinado con un test de carga, valida la resiliencia real de tu sistema.

Caso: añadir 500ms de latencia a un servicio durante 30s

import http from 'k6/http';
import { ServiceDisruptor } from 'k6/x/disruptor';

export const options = {
    scenarios: {
        // Genera tráfico normal
        load: {
            executor: 'constant-arrival-rate',
            rate: 100, timeUnit: '1s',
            duration: '2m', preAllocatedVUs: 50,
            exec: 'normalLoad',
        },
        // En paralelo, inyecta fallas tras 30s
        chaos: {
            executor: 'shared-iterations',
            iterations: 1, vus: 1,
            startTime: '30s',
            exec: 'injectFault',
        },
    },
    thresholds: {
        // El SLO debe mantenerse incluso bajo caos
        http_req_duration: ['p(95) < 1500'],
        http_req_failed:   ['rate < 0.10'],
    },
};

export function normalLoad() {
    http.get('http://api-service.default.svc.cluster.local/users');
}

export function injectFault() {
    const disruptor = new ServiceDisruptor('api-service', 'default');
    disruptor.injectHTTPFaults(
        {
            averageDelay: '500ms',     // +500ms a todas las requests
            errorRate:    0.1,         // 10% devuelve error
            errorCode:    503,
        },
        '30s'    // duración de la inyección
    );
}

import { PodDisruptor } from 'k6/x/disruptor';

export function chaos() {
    const pd = new PodDisruptor({ namespace: 'default',
                                  selector: { labels: { app: 'api' } } });
    pd.terminatePods({ count: 1 });   // mata 1 pod
    // o por proporción

🏆 Mejores prácticas y antipatrones a evitar

✅ Hacer (Do's)

  • Usa SharedArray para datasets grandes — una sola copia en memoria.
  • Mantén los scripts idempotentes: que correr el test dos veces no rompa nada.
  • Define thresholds basados en SLOs reales (ej: p95 latencia < 300ms).
  • Parametriza por ambiente con __ENV. Nunca hardcodees URLs ni credenciales.
  • Usa group() para agrupar flujos lógicos y obtener métricas separadas.
  • Inyecta sleep() realistas (think time) — no es throughput máximo lo que mides en load tests.
  • Tagea peticiones para poder filtrar métricas ({ tags: { endpoint: 'login' } }).
  • Modulariza: separa configuración, helpers, flujos de negocio.
  • Usa __VU + __ITER para variabilidad determinista (reproducible).
  • Versiona tus scripts en Git junto al código de la aplicación.
  • Empieza con smoke, luego load, luego stress, en pipelines separados.

❌ Evitar (Don'ts)

  • No uses Math.random() sin control — datos no reproducibles entre runs.
  • No proceses JSON enormes o cálculos pesados dentro de default() — sesga las métricas.
  • No lances pruebas de carga contra producción sin acuerdo del equipo.
  • No corras contra entornos compartidos sin coordinación.
  • No abuses de console.log en producción del test — degrada el rendimiento.
  • No mezcles autenticación con lógica de negocio en el mismo archivo.
  • No olvides el teardown que limpia datos creados.
  • No infieras conclusiones de un solo run — siempre repite y compara.
  • No ignores http_req_blocked: si es alto, tu test está limitado por el pool de conexiones, no por el servidor.

import { check, sleep, group } from 'k6';
import { SharedArray } from 'k6/data';
import { Rate, Trend } from 'k6/metrics';
import { uuidv4, randomItem } from 'https://jslib.k6.io/k6-utils/1.4.0/index.js';

// ── Métricas custom ───────────────────────────────────────
const errorRate    = new Rate('error_rate');
const customTrend  = new Trend('checkout_total', true);

// ── Data en memoria compartida ────────────────────────────
const productos = new SharedArray('productos', () =>
    JSON.parse(open('./data/productos.json'))
);

// ── Configuración ─────────────────────────────────────────
export const options = {
    scenarios: {
        load: {
            executor: 'ramping-vus',
            startVUs: 0,
            stages: [
                { duration: '1m', target: 20 },
                { duration: '3m', target: 20 },
                { duration: '30s', target: 0 },
            ],
            exec: 'flowCheckout',
            tags: { test_type: 'load' },
        },
    },
    thresholds: {
        error_rate:            ['rate < 0.05'],
        checkout_total:        ['p(95) < 800'],
        'http_req_duration{endpoint:checkout}': ['p(95) < 600'],
        'group_duration{group:::Checkout flow}': ['p(95) < 1500'],
    },
};

// ── Setup global ──────────────────────────────────────────
export function setup() {
    const res = http.post(`${__ENV.BASE_URL}/auth/login`, JSON.stringify({
        user: __ENV.USER,  pass: __ENV.PASS,
    }), { headers: { 'Content-Type': 'application/json' } });
    return { token: res.json('token') };
}

// ── Flujo principal ───────────────────────────────────────
export function flowCheckout(data) {
    const headers = {
        Authorization: `Bearer ${data.token}`,
        'X-Request-Id': uuidv4(),
        'Content-Type': 'application/json',
    };

    group('Checkout flow', () => {
        let res;
        group('Add to cart', () => {
            const producto = randomItem(productos);
            res = http.post(`${__ENV.BASE_URL}/cart`,
                JSON.stringify({ productId: producto.id, qty: 1 }),
                { headers, tags: { endpoint: 'cart' } });
            check(res, { 'cart 201': (r) => r.status === 201 });
            errorRate.add(res.status !== 201);
        });

        const start = Date.now();
        group('Checkout', () => {
            res = http.post(`${__ENV.BASE_URL}/checkout`,
                JSON.stringify({ payment: 'card' }),
                { headers, tags: { endpoint: 'checkout' } });
            check(res, { 'checkout 200': (r) => r.status === 200 });
            errorRate.add(res.status !== 200);
        });
        customTrend.add(Date.now() - start);
    });

    sleep(Math.random() * 3 + 1);   // think time 1-4s
}

export function teardown(data) {
    console.log('Test completado, limpiando con token:', data.token.slice(0, 10) + '...');
}

3. HTTP client con retries y observabilidad

// helpers/http-client.ts
import http from 'k6/http';

const DEFAULT_HEADERS = { 'Content-Type': 'application/json', 'X-Client': 'k6-tests' };

export function request(method, url, body, opts = {}) {
    const params = {
        headers: { ...DEFAULT_HEADERS, ...(opts.headers || {}) },
        tags:    opts.tags || {},
        timeout: opts.timeout || '30s',
    };
    let res, attempt = 0;
    const maxRetries = opts.retries || 0;
    do {
        res = http.request(method, url, body, params);
        if (res.status < 500 || attempt >= maxRetries) break;
        attempt++;
    } while (true);
    return res;
}

Cuándo el problema es del test, no de la app

Heurísticas:
  • Si http_req_waiting es bajo pero http_req_duration es alto → red o cliente saturado, NO el servidor.
  • Si pones --vus 1 --duration 30s y todavía hay errores → el problema NO es de carga.
  • Si los errores aparecen exactamente cuando arranca el spike, mira en logs del balanceador.
  • Si la CPU del cliente k6 está al 100% → necesitas distribuir (k6-operator o cloud).
  • Si el throughput tope no escala con VUs → estás en VU-bound; cambia a arrival-rate.
